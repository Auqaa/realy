# QR Photo Scan Reliability

## Problem Frame

Сейчас QR-поток в мобильном браузере частично работает: backend принимает валидные коды, а ручной ввод успешен, но сценарий "сделать фото / use photo" даёт сбой при распознавании QR. Это особенно заметно на iPhone, где пользователь нажимает кнопку открытия камеры телефона, делает снимок и получает ошибку распознавания вместо успешного сканирования.

Проблема не в самих QR-значениях и не в API обработки сканов: ручной ввод того же кода проходит. Значит продуктовая задача — сделать photo-based QR scan надёжным в веб-клиенте на мобильных устройствах без изменения базового UX.

## Requirements

**Existing UX**
- R1. Текущая кнопка мобильного сканирования должна сохраниться как основной вход в photo-based QR flow.
- R2. Пользователь не должен быть вынужден выбирать другой сценарий взаимодействия только из-за платформы; flow должен одинаково работать на iPhone и Android.

**Photo Recognition Reliability**
- R3. После съёмки или выбора фото с QR приложение должно пытаться надёжно распознать код на iPhone и Android.
- R4. Если фото содержит валидный и читаемый QR-код, приложение должно переходить к обычной обработке скана без дополнительного ручного вмешательства.
- R5. Решение должно учитывать типичные mobile-specific особенности снимков: ориентацию, размер изображения, формат файла и особенности browser capture flow.

**Failure Handling**
- R6. Если QR действительно не удаётся прочитать, сообщение об ошибке должно оставаться понятным и не вводить пользователя в заблуждение.
- R7. При неудачном распознавании пользователь должен иметь возможность сразу повторить попытку тем же экраном, без сложного восстановления состояния.
- R8. Ручной ввод кода остаётся доступным как fallback, но не считается основным путём для нормального мобильного сценария.

## Success Criteria

- На iPhone сценарий "открыть камеру телефона → сделать фото → use photo" успешно распознаёт рабочий QR-код.
- На Android тот же photo-based flow тоже успешно работает.
- Ошибка распознавания появляется только на реально плохих/нечитаемых снимках, а не на обычных валидных фото QR.
- Пользовательский flow визуально остаётся почти тем же, без лишнего усложнения интерфейса.

## Scope Boundaries

- Не меняем базовый UX на отдельный многошаговый QR-wizard.
- Не уводим пользователей в обязательный live-camera-only режим.
- Не переносим QR-распознавание на backend как основной OCR/vision pipeline без крайней необходимости.
- Не меняем server-side scan contract, если проблема остаётся чисто client-side.

## Key Decisions

- Сохраняем текущую одну кнопку mobile scan, потому что проблема в надёжности реализации, а не в выбранной UX-модели.
- Считаем ручной ввод fallback-механикой, а не приемлемой заменой исправлению photo flow.
- Целевой результат — кроссплатформенная стабильность именно в mobile web, а не workaround только для одной платформы.

## Dependencies / Assumptions

- Текущий backend scan flow корректен, потому что ручной ввод QR-значения проходит успешно.
- Основная зона проблемы находится в frontend QR photo-decoding path.
